==============================================================================
  SECTION 1: ARCHITECTURE OVERVIEW
==============================================================================

CLASHCODE is a competitive coding platform deployed using a microservices
architecture on AWS EC2 using K3s (Lightweight Kubernetes).

  Internet
     |
     | HTTPS / WSS (Port 443)
     v
  [Traefik Ingress Controller]  <--  cert-manager (Let's Encrypt SSL)
  (api-coc.jithin.site)
     |
     |-- /api/*           --> [core] Django (Port 8000)
     |-- /                --> [core] Django (Port 8000)
     |-- /ws/chat         --> [chat] FastAPI WebSocket (Port 8001)
     |-- /ws/notifications--> [chat] FastAPI WebSocket (Port 8001)
     |-- /ai/*            --> [ai]   FastAPI AI Service (Port 8002)
                                         |
                          [vector-db] ChromaDB (Port 8000, internal)

  Background Jobs:
     [celery-worker]  <-- reads from [core] image, Redis broker
     [celery-beat]    <-- Django DB scheduler

  External Services (not self-hosted):
     - Supabase (PostgreSQL) — Primary database
     - Upstash (Redis) — Cache, Pub/Sub, OTP throttle, Celery broker
     - AWS DynamoDB — Chat message history
     - Cloudinary — User media & image uploads
     - Groq API — LLM inference for AI hints
     - AWS SES — Transactional emails

==============================================================================
  SECTION 6: K3S INSTALLATION (LIGHTWEIGHT KUBERNETES)
==============================================================================

  K3s is a production-grade, lightweight Kubernetes distribution by Rancher.
  It comes pre-installed with Traefik as the Ingress Controller.

  STEP-BY-STEP INSTALLATION ON EC2:
  -----------------------------------

  # 1. Update the OS
  sudo apt-get update && sudo apt-get upgrade -y

  # 2. Install K3s
  curl -sfL https://get.k3s.io | sh -

  # 3. Wait ~30 seconds, then verify the node is ready
  sudo kubectl get nodes
  # Should show: STATUS = Ready

  # 4. Allow non-root kubectl access
  sudo chmod 644 /etc/rancher/k3s/k3s.yaml

  # 5. Create the application namespace
  sudo kubectl create namespace coc

  KEY K3s FACTS:
  - Default ingress:  Traefik (available automatically)
  - Config location:  /etc/rancher/k3s/k3s.yaml
  - kubeconfig:       K3s writes it automatically at /etc/rancher/k3s/k3s.yaml

==============================================================================
  SECTION 8: GITHUB ACTIONS CI/CD PIPELINE
==============================================================================

  Location: services/.github/workflows/deploy.yml

  WHAT IT DOES:
  - Triggered on every push to the "main" branch
  - Builds three Docker images (core, chat, ai)
  - Pushes them to GHCR with the tag ":latest"

  WORKFLOW STRUCTURE:
  -------------------
  name: Build and Push to GHCR

  on:
    push:
      branches: [ "main" ]

  env:
    REGISTRY: ghcr.io
    IMAGE_NAME: ${{ github.repository }}

  jobs:
    build-and-push:
      runs-on: ubuntu-latest
      permissions:
        contents: read
        packages: write
      steps:
        - uses: actions/checkout@v4
        - uses: docker/login-action@v3
          with:
            registry: ghcr.io
            username: ${{ github.actor }}
            password: ${{ secrets.GITHUB_TOKEN }}
        - uses: docker/build-push-action@v5  # for core
          with: { context: ./services/core, push: true, tags: ...core:latest }
        - uses: docker/build-push-action@v5  # for chat
          with: { context: ./services/chat, push: true, tags: ...chat:latest }
        - uses: docker/build-push-action@v5  # for ai
          with: { context: ./services/ai, push: true, tags: ...ai:latest }

  TO TRIGGER A REDEPLOY:
  - Push any commit to "main" → images rebuild automatically
  - Then on EC2, run:
    sudo kubectl rollout restart deployment core -n coc
    sudo kubectl rollout restart deployment chat -n coc
    sudo kubectl rollout restart deployment ai -n coc
    sudo kubectl rollout restart deployment celery-worker -n coc
    sudo kubectl rollout restart deployment celery-beat -n coc

  Location: services/deploy/k3s/

  FILES:

    --from-literal=CLOUDINARY_API_SECRET="your-api-secret" \

==============================================================================
  SECTION 10: HTTPS & SSL WITH LET'S ENCRYPT (CERT-MANAGER)
==============================================================================

  cert-manager automates requesting and renewing TLS certificates from
  Let's Encrypt. It runs inside the K3s cluster.

  STEP 1: Install cert-manager
  ----------------------------
  kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.13.0/cert-manager.yaml

  STEP 2: Create a ClusterIssuer
  --------------------------------
  Create a file named issuer.yaml on the server:

    apiVersion: cert-manager.io/v1
    kind: ClusterIssuer
    metadata:
      name: letsencrypt-prod
    spec:
      acme:
        server: https://acme-v02.api.letsencrypt.org/directory
        email: jzdieheart@gmail.com
        privateKeySecretRef:
          name: letsencrypt-prod
        solvers:
        - http01:
            ingress:
              class: traefik

  Apply: kubectl apply -f issuer.yaml

  STEP 3: Verify certificate was issued
  ----------------------------------------
  kubectl get certificate -n coc
  # Should show: READY = True

  The ingress.yaml already contains the annotation:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
  ...so new TLS certs are requested automatically when the ingress is applied.


==============================================================================
  SECTION 11: INGRESS ROUTING (TRAEFIK)
==============================================================================

  Traefik is the default Ingress Controller bundled with K3s.
  It handles all external HTTP/HTTPS traffic and routes it to internal pods.

  ROUTING TABLE (api-coc.jithin.site):
  ---------------------------------------
  Path Prefix        Backend Service    Backend Port
  /ws/chat           chat               8001
  /ws/notifications  chat               8001
  /api/*             core               8000
  /ai/*              ai                 8002  (with strip-prefix middleware)
  /                  core               8000

  MIDDLEWARE (middlewares.yaml):
  --------------------------------
  A "strip-ai-prefix" middleware removes the /ai prefix before forwarding
  requests to the AI service, so it receives clean paths like /health, /hints.

  TLS (HTTPS):
  - Both ingress objects request a TLS cert for api-coc.jithin.site
  - Secret name: coc-api-tls (auto-created by cert-manager)
  - Supports both web (HTTP) and websecure (HTTPS) entrypoints

     Table name: ChatMessages

  - Handled via WhiteNoise middleware (not a separate Nginx sidecar)
  - WhiteNoise is now in requirements.txt for the core service
  - collectstatic is run inside the migrate job
  - WhiteNoiseMiddleware is registered in Django settings (after SecurityMiddleware)
  - Storage: CompressedManifestStaticFilesStorage

==============================================================================
  SECTION 16: TROUBLESHOOTING & KNOWN ISSUES
==============================================================================

  ISSUE: OOMKilled (Out of Memory)
  ---------------------------------
  Cause:   Memory limits too tight (original: 256Mi for core).
  Fix:     Increased to 512Mi for core, 800Mi for ai in deployments.yaml.
  Also:    Add a 1GB swap file on the EC2 instance (see Section 4).

  ISSUE: DNS Resolution failure for Supabase
  -------------------------------------------
  Cause:   K3s internal DNS (CoreDNS) may fail to resolve external hostnames.
  Fix:     Use the IP-based Supabase connection string, OR check CoreDNS pods.
           sudo kubectl get pods -n kube-system
           sudo kubectl logs -l app=coredns -n kube-system

  ISSUE: ChromaDB vector data lost after restart
  -----------------------------------------------
  Cause:   vector-db pod had no persistent volume.
  Fix:     Added pvc.yaml (5GiB PVC) mounted at /chroma/chroma.

  ISSUE: AI service 404 (routes not found)
  -----------------------------------------
  Cause:   Original ingress.yaml had no rule for /ai/* path.
  Fix:     Added separate ingress object for AI service with strip-prefix
           middleware so the AI service receives correct paths.

  ISSUE: Images can't be pulled (ImagePullBackOff)
  --------------------------------------------------
  Cause:   GHCR secret not created or not referenced in deployment.
  Fix:     Create ghcr-secret (see Section 7) and ensure it is referenced.

  ISSUE: Static files returning 404
  -----------------------------------
  Cause:   No Nginx to serve Django static files.
  Fix:     WhiteNoise middleware added to Django settings. collectstatic
           runs in the migrate job.

  ISSUE: JWT authentication failing
  -----------------------------------
  Cause:   RS256 private/public key passed as single-line env var (\\n issue).
  Fix:     The Django settings.py includes .replace("\\n", "\n") on the key.
           When creating as K8s secret, use --from-literal with actual newlines
           or --from-file with a .pem file.

  ISSUE: Celery beat / worker not picking up tasks
  -------------------------------------------------
  Cause:   REDIS_URL not available or Celery app not loading correctly.
  Fix:     Verify REDIS_URL in secrets → kubectl exec core -- env | grep REDIS


==============================================================================
  SECTION 17: REDEPLOYMENT WORKFLOW (UPDATE STEPS)
==============================================================================

  NORMAL CODE UPDATE:
  --------------------
  1. Push code changes to the "main" branch of the GitHub repo.
  2. GitHub Actions automatically builds and pushes new images to GHCR.
  3. On the EC2 server, restart the relevant deployments:

     sudo kubectl rollout restart deployment core -n coc
     sudo kubectl rollout restart deployment chat -n coc
     sudo kubectl rollout restart deployment ai -n coc
     sudo kubectl rollout restart deployment celery-worker -n coc
     sudo kubectl rollout restart deployment celery-beat -n coc

  4. Monitor rollout:
     sudo kubectl rollout status deployment/core -n coc

  UPDATING CONFIG/SECRETS (without code change):
  ------------------------------------------------
  # Update configmap:
  sudo kubectl edit configmap coc-config -n coc
  # Then restart affected pods:
  sudo kubectl rollout restart deployment core -n coc

  # Update a specific secret value:
  sudo kubectl patch secret coc-secrets -n coc \
    --type='json' \
    -p='[{"op":"replace","path":"/data/DB_PASSWORD","value":"'$(echo -n new_pwd | base64)'"}]'

  FULL FRESH REDEPLOY (nuclear option):
  ---------------------------------------
  sudo kubectl delete all --all -n coc
  sudo kubectl apply -f pvc.yaml
  sudo kubectl apply -f services.yaml
  sudo kubectl apply -f migrate-job.yaml
  # Wait for migrate to complete
  sudo kubectl apply -f deployments.yaml
  sudo kubectl apply -f middlewares.yaml
  sudo kubectl apply -f ingress.yaml

  VERIFY EVERYTHING IS WORKING:
  --------------------------------
  curl https://api-coc.jithin.site/health/
  # Expected: {"status": "ok", ...}

  curl https://api-coc.jithin.site/ai/health
  # Expected: {"status": "ok"}
